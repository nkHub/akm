# Markdown KB 自动同步 Skill 草案

## 目标

在不修改 AKM 核心代码的前提下，设计一个可复用的 skill，用于把本地 Markdown 文档同步到 `markdown_kb` 的全局知识库源目录，并在同步完成后触发索引更新。

第一版只关注文档同步编排，不扩展知识库检索策略，也不默认拦截普通聊天请求。

## 设计边界

这份草案只基于当前仓库中已经存在的 `markdown_kb` 设计文档与 README 描述整理，目的是先把 skill 的职责、输入输出和执行流程沉淀下来。

当前仍有以下边界需要明确：

- 真实 skill 宿主规范尚未补全，例如 skill 的发现、注册与触发细节
- `markdown_kb` 相关接口虽然在文档中已被提及，但请求参数与响应格式仍需结合实现进一步核实
- 这份文档不假设“上传动作本身就会自动重建索引”
- 本文把“全局知识库源目录”作为概念目标来描述，但当前仓库实现中的实际目录名仍需以代码为准

因此，本文档是一个面向实现前讨论的草案，而不是最终定稿。

## 全局知识库源目录

这份 skill 草案建议把 `markdown_kb` 当前使用的文档目录视为一个全局知识库源目录，skill 的核心职责不再是“把文件传给某个临时上传入口”，而是“把外部 Markdown 文档收敛进这一个全局目录，再驱动索引刷新”。

如果按你的目标描述，可以把这个概念目录写成 `~/.akm/markdown-kb/docs`。

但需要明确一件事：当前仓库实现里，`markdown_kb` 实际使用的目录路径是 `~/.akm/markdown_kb/docs`，也就是目录名中使用下划线 `_`，不是连字符 `-`。这一点已经在插件代码中固定：

- 数据根目录是 `~/.akm/markdown_kb/`
- 原始 Markdown 文档目录是 `~/.akm/markdown_kb/docs`

当前插件页面的状态区已经直接展示了 `docs_dir`，也就是插件当前实际使用的文档目录。这个字段来自 `/api/markdown-kb/status` 的返回值，页面上显示为“文档目录”。

但需要特别注意：当前这里是“状态展示”，不是 `plugin.json` 里的用户可配置设置项。也就是说：

- 现阶段页面会告诉用户当前文档目录在哪里
- 但用户不能通过插件设置表单直接改这个目录

因此，后续如果把这份草案落地成实际 skill，建议遵循以下约定：

1. 在产品描述层，可以把它称为“全局知识库源目录”。
2. 在实现层，应优先读取 `/api/markdown-kb/status` 返回的 `docs_dir`，并以此作为真实目标目录。
3. 如果拿不到状态接口结果，再回退到当前默认目录 `~/.akm/markdown_kb/docs`。
4. 除非后续代码已经统一迁移到 `~/.akm/markdown-kb/docs`，否则文档里不应把连字符路径写成当前既定事实。

## Skill 定位

建议将该 skill 定义为一个“全局知识库文档同步编排器”，而不是知识库核心功能本身。它只负责串联现有能力：

- 收集待同步的 `.md` 文件
- 把文档同步到全局知识库源目录
- 调用 `markdown_kb` 的同步或重建能力刷新索引
- 调用状态或健康检查接口校验最终结果

这样做的好处是：

1. 不需要侵入 AKM 核心链路。
2. 不需要把上传、索引、状态检查逻辑分散在多个手工步骤里。
3. 能够把“文档进入全局知识库源目录后还要记得更新索引”这个流程变成一次显式操作。

## 适用场景

这个 skill 适合处理以下场景：

- 新增了一批 `.md` 文档，需要统一入库
- 修改了已有 Markdown 文档，需要把变更同步到知识库
- 希望把 `markdown_kb` 的 docs 目录当作一个全局文档池统一维护
- 需要在上传后立即刷新检索索引，保证问答链路读取到最新文档
- 需要对知识库进行一次显式的增量同步或全量重建

## 输入

支持以下任一输入方式：

1. 单个 Markdown 文件路径
2. 多个 Markdown 文件路径
3. 一个目录路径，由 skill 自动扫描其中的 `.md` 文件

默认目标目录建议指向全局知识库源目录。具体解析顺序建议如下：

1. 优先读取 `/api/markdown-kb/status` 返回的 `docs_dir`
2. 如果状态接口不可用，再回退到当前实现里的 `~/.akm/markdown_kb/docs`

建议支持以下可选参数：

- `mode`: `sync` / `rebuild`
- `dry_run`: 是否只预览待处理文件，不真正上传
- `verbose`: 是否输出详细执行日志

如果后续把这个 skill 真正落地，建议再补一个可选参数：

- `write_mode`: `direct` / `upload_api`

参数语义建议如下：

- `sync`：优先用于日常增量更新
- `rebuild`：优先用于大批量变更、首次完整导入，或怀疑索引状态不一致时
- `dry_run`：只输出待处理文件清单与预计动作，便于用户确认范围
- `write_mode`：默认 `direct`，表示直接维护 `docs_dir` 下的文件；`upload_api` 只作为兼容现有页面上传入口的辅助模式

## 初始化知识模块

如果 skill 需要支持“先初始化一份项目知识库”，建议默认生成一份固定五模块结构的知识文档，而不是硬编码目录模板或自由散写概述。

建议规则如下：

1. 初始化文档目标文件名默认直接使用项目名称，即 `<project_name>.md`。
2. 初始化内容应尽量来自当前仓库中可验证的 README、设计文档和关键代码结构。
3. 若资料不足以支持某个细节，应明确写“资料中未提及”，而不是补出臆测内容。
4. 初始化文档应固定拆成 `P1 方法论`、`P2 问题解决方案`、`P3 概念原理`、`P4 外部知识精炼`、`P5 关联映射` 五个模块，并尽量条目化写作。
5. 初始化动作只创建知识文档，不创建目录树模板。

## 输出

执行完成后，建议 skill 返回以下结果：

1. 本次发现的 Markdown 文件列表
2. 计划同步到全局知识库源目录的文件列表
3. 成功写入全局知识库源目录的文件列表
4. 同步失败的文件列表与失败原因
5. 本次采用的索引更新方式
6. 索引更新结果
7. 最终状态检查结果

如果输入为空、目录中没有 `.md` 文件，或预检查失败，也应返回明确、可操作的错误信息。

## 执行流程

### 1. 收集目标文件

执行时先根据输入类型决定收集方式：

- 如果输入是单个文件，则校验扩展名必须为 `.md`
- 如果输入是多个文件，则逐个筛选出合法 Markdown 文件
- 如果输入是目录，则递归扫描目录中的 `.md` 文件

如果没有发现任何 `.md` 文件，应立即终止并提示用户，而不是继续进入上传阶段。

在这个阶段，还应明确源目录与目标目录的角色：

- 源目录：用户本次希望同步的本地目录或文件集合
- 目标目录：插件状态接口暴露出来的 `docs_dir`，若不可用则回退到当前默认目录 `~/.akm/markdown_kb/docs`

这意味着 skill 的本质动作更接近“同步入全局知识库源目录”，而不是“直接把源目录当作索引目录使用”。

### 2. 预检查

在执行上传前，先检查以下条件：

1. `markdown_kb` 插件是否已启用并可访问
2. 插件状态接口返回的 `docs_dir` 是否可用
3. 索引服务是否可用
4. 必需配置是否已经完成

基于当前仓库描述，至少要特别关注：

- `embedding_model` 是否已配置，因为这是当前插件链路中的必填项
- 如果要执行问答或重排相关验证，再额外检查 `chat_model` 与可选 `reranker_model`

如果任一检查失败，则停止执行并返回原因。

### 3. 维护文档目录

这里更准确地说，应当是“维护全局知识库源目录中的文件”，而不只是“上传”。

同步阶段建议只处理 `.md` 文件，不修改文件内容。

执行策略建议如下：

- 支持批量同步多个 Markdown 文件
- 对每个文件记录成功或失败结果
- 对失败项保留错误信息，便于后续单独重试
- 同步完成后，全局知识库源目录应该成为后续 `sync / rebuild` 的唯一标准输入目录

推荐把“直接维护 `docs_dir` 中的文件”作为主路径：

1. 直接把新的 `.md` 文件写入 `docs_dir`
2. 直接修改 `docs_dir` 中已有的 `.md` 文件
3. 直接删除 `docs_dir` 中不再需要的 `.md` 文件
4. 完成后再执行 `sync` 或 `rebuild`

这样更贴近当前实现，因为当前索引刷新逻辑本身就是直接扫描 `docs_dir` 来判断新增、变化和删除。

上传接口应被视为辅助入口，而不是唯一入口。也就是说：

- 如果用户已经在本地脚本、编辑器或其他流程里直接维护 `docs_dir`
- 那么 skill 不需要再把这些文件“重新上传一遍”
- 此时只要补上状态检查与索引更新即可

根据当前 README 中对插件能力的描述，上传页面已经支持一次选择多个 `.md` 文件批量保存，但上传阶段本身仍不会自动重建索引。因此 skill 不应把“文件已经进入全局知识库源目录”等同于“索引已更新”。

需要额外说明的是，当前实现会把上传文件保存成扁平的 basename，并且同名文件会直接覆盖旧文件。因此如果未来真的把该目录作为“全局知识库源目录”长期维护，就必须正视同名文件冲突问题。这是当前实现的已知限制，skill 文档里应当显式提示，而不能假设同名文件可以自然并存。

另外，既然页面状态区已经能展示“文档目录”，skill 在执行前也应优先以这个运行时状态为准，而不是把文档中写死的默认路径当作唯一真相。

### 4. 更新索引

完成目录维护后，再显式执行索引更新。

建议约定如下：

- 当 `mode=sync` 时，执行一次增量同步
- 当 `mode=rebuild` 时，执行一次全量重建
- 如果用户未指定 `mode`，默认使用 `sync`

这样设计的原因是：

- 增量同步更适合全局知识库源目录中的日常文档变更
- 全量重建更适合首次导入、大范围调整，或索引与原文状态明显不一致时

如果后续确认插件已经提供单文件重建能力，也可以补充一种“单文件上传后仅重建该文件”的细粒度模式；但第一版不必预设这个抽象，避免过早复杂化。

如果当前 skill 采用的是“直接维护 `docs_dir`”模式，那么这里的动作本质上就是：

- 先把 `docs_dir` 调整到目标状态
- 再让 `markdown_kb` 基于这个目录状态更新索引

这是当前实现最自然、也最贴近代码真实行为的工作流。

## 数据一致性说明

为了避免把“文件写入文档目录”和“文件已经进入索引数据库”混为一谈，这里需要单独说明当前实现的数据分层。

### 1. 原始文档层

`docs_dir` 保存的是 Markdown 原文文件。无论是通过上传页面写入，还是直接在本地维护这个目录，本质上都是在维护原始文档层。

这意味着：

- 上传动作本身只负责把文件写入 `docs_dir`
- 直接维护 `docs_dir` 不会因为“没走上传接口”就丢失原始文档
- 只要文件已经成功写入这个目录，原文就已经被保留下来了

### 2. 索引数据库层

索引数据库保存的是切片结果、向量和索引元数据。它不会因为文档目录发生变化而自动实时刷新，仍然需要显式执行 `sync` 或 `rebuild`。

这意味着：

- 上传动作不会自动把文档写入索引数据库
- 直接维护 `docs_dir` 后，如果不执行索引更新，数据库内容可能仍然是旧状态
- skill 不应把“文件已经在目录里”解释成“检索一定能命中新内容”

### 3. 可能出现的漂移场景

如果只维护 `docs_dir` 而不更新索引，会出现以下漂移：

- 新增了 Markdown 文件，但检索还命不中，因为它尚未入索引
- 修改了 Markdown 文件，但检索或问答仍然基于旧切片和旧向量
- 删除了 Markdown 文件，但索引数据库里可能还残留旧 chunks，直到后续同步清理

所以“直接维护文档目录”本身不是风险，真正的风险是“维护完目录后忘了更新索引”。

### 4. Skill 应承担的职责

如果这个 skill 以“直接维护 `docs_dir`”为主路径，那么它至少应保证两件事：

1. 把目标文件集正确写入或同步到 `docs_dir`
2. 在目录调整完成后，显式执行一次 `sync` 或 `rebuild`

只有把这两个步骤串起来，skill 才能真正保证“原文目录状态”和“索引数据库状态”尽量保持一致。

### 5. 状态校验

索引更新后，再执行一次状态或健康检查，确认：

1. 服务仍然可用
2. 索引更新流程已执行完成
3. 没有明显失败项遗留

如果当前插件的状态接口已经提供文件级结果清单，则建议 skill 直接把这些清单展示出来，而不是只输出摘要数字。

## 默认触发意图

建议让该 skill 匹配以下显式意图：

- “把这个 docs 目录同步到 markdown_kb”
- “把这些文档同步到全局知识库”
- “把本地 markdown 收进 markdown_kb 的全局 docs 目录”
- “上传这些 markdown 并更新索引”
- “重建 markdown_kb 索引”
- “同步知识库文档”

不建议默认拦截所有与 Markdown 相关的对话，因为这会让 skill 行为过于隐式，也不符合当前 `markdown_kb` 偏显式操作的设计方向。

## 失败处理策略

### 上传失败

- 返回失败文件列表
- 不自动删除已成功写入全局知识库源目录的文件
- 允许用户只重试失败项

这里的“上传失败”在 `write_mode=direct` 模式下，更准确地说应理解为“写入或复制到 `docs_dir` 失败”。

### 索引更新失败

- 保留全局知识库源目录中的同步结果
- 返回索引更新失败信息
- 建议用户改用 `rebuild` 模式重试

### 无可处理文件

- 明确提示未发现 `.md` 文件
- 不执行任何上传或索引动作

## 安全约束

第一版建议遵守以下约束：

- 只处理 `.md` 文件
- 不修改源 Markdown 文件内容
- 默认只向全局知识库源目录写入，不直接改动索引数据库文件
- 不删除远端已有文档，除非用户明确要求
- 不在未确认目标路径时扫描整个工作区
- 对“同名文件冲突”给出显式提示，不假设不同来源的同名文件可以共存

这些约束可以把 skill 控制在“同步器”范围内，避免它演变成一个拥有过多副作用的文档管理器。

## 推荐操作示例

下面这几种工作流最贴近当前 `markdown_kb` 的真实实现，也最适合作为后续 skill 的默认行为模板。

### 1. 日常维护后执行 `sync`

适用场景：

- 新增了少量 Markdown 文档
- 修改了现有文档内容
- 删除了少量过期文档

推荐步骤：

1. 直接维护 `docs_dir` 中的 `.md` 文件
2. 执行一次 `sync`
3. 查看状态或健康检查结果，确认新增 / 变化 / 删除已经被正确吸收

这是最适合日常使用的默认工作流，因为它更轻量，也更符合当前增量同步的设计目标。

### 2. 批量替换后执行 `rebuild`

适用场景：

- 一次性替换了一大批文档
- 调整了文档切片相关配置
- 怀疑索引数据库和原始文档目录已经出现明显漂移

推荐步骤：

1. 先把 `docs_dir` 调整到目标状态
2. 直接执行一次 `rebuild`
3. 重建完成后再查看状态和检索结果

这种方式成本更高，但更适合在大范围变更后恢复“目录状态”和“索引状态”的整体一致性。

### 3. 处理同名文件时先重命名再同步

适用场景：

- 来自不同来源的文档拥有相同文件名
- 准备把多个目录中的文档收敛进同一个全局知识库源目录

推荐步骤：

1. 在写入 `docs_dir` 前先做文件名去冲突
2. 确保全局知识库源目录中每个 `.md` 文件名唯一
3. 再执行 `sync` 或 `rebuild`

原因是当前实现把文档目录当作扁平文件池处理，并且同名文件会互相覆盖。因此 skill 如果要长期稳定工作，应该把“避免同名冲突”当成默认前置规则，而不是事后补救动作。

## 最小可行版本

如果只做最小可行版本，建议先实现以下能力：

1. 输入一个目录或文件列表
2. 过滤出 `.md` 文件
3. 直接同步到 `docs_dir`
4. 执行 `sync`
5. 执行状态检查
6. 返回结果摘要

这个版本已经足以满足“直接维护文档目录并更新文档索引”的核心目标。
